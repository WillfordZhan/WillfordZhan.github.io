别再让 Agent 靠感觉停下来了：聊聊通用多跳规划能力怎么做

最近在调一条生产计划问答链时，碰到一个很典型的问题。

用户问的是“今天计划炉次有多少”，系统却在第一跳 production_plan_search 后就停了，直接把计划列表里的 count/total 当成了答案。真正需要的链路其实是：

  1. 先查计划，定位目标计划。
  2. 再查计划详情。
  3. 最后基于详情里的计划炉次做汇总。

这个问题看起来像“模型不够聪明”，但本质不是。更准确地说，是系统没有把“什么时候可以停，什么时候必须继续查”做成显式能力。

为什么 Agent 会在第一跳就停下来

这类问题一般不是单点故障，而是几层因素叠在一起。

第一层是 tool 本身的语义边界不够清楚。

如果一个工具叫 production_plan_search，返回里又有 count、total、items，模型很容易把它理解成“已经拿到了足够的统计结果”。但实际上，这个工具只能回答“有几个计划”“有哪些计划”，并不能回答“某个计划里有多少个计划炉次”。

第二层是 planner 的停机条件太宽。

很多系统都会给模型一条类似的规则：如果证据已经足够，就直接回答，不要继续调工具。这条规则本身没错，问题在于“证据是否足够”如果完全交给模型自己判断，它往往会选择最短路径，而不是最稳路径。

第三层是缺少“证据完成条件”。

对于“计划炉次有多少”这类问题，真正的回答证据应该来自计划详情，而不是计划列表。也就是说，这类问题的完成条件不是“拿到 search result”，而是“拿到 detail result 里的炉次字段或炉次计数”。如果系统没有把这个完成条件显式表达出来，模型就会用自己看到的第一份结构化结果尽快收尾。

这不是某一句 query 的问题

如果只盯着“今天计划炉次有多少”这一句，很容易走到一个错误方向：给这句话写一条 if/else，或者在 prompt 里专门补一条规则。

这种做法短期能压住一个 case，但很快会失效。因为用户下一次可能会问：

  • 今天这个计划的炉次安排数量是多少
  • 这个生产计划下有几炉
  • 当前计划一共排了多少个炉次
  • 今天 9 号炉的计划里现在有几炉

这些问题在业务上是同一种意图，但文本不一样。如果靠 query patch，一定会越写越碎。

所以更合理的做法不是“修一句话”，而是把多跳规划能力做成系统能力。

业界一般怎么增强这种通用多跳规划能力

如果把官方文档和企业 Agent 实践放在一起看，主流方法基本都落在四件事上。

1. 先把 tool 的职责边界说清楚

这是最基础的一层。

一个工具要明确：

  • 它负责做什么
  • 不负责做什么
  • 它更像“定位工具”还是“证据工具”
  • 如果用户要问更深一层的问题，通常应该接哪个工具

比如：

  • production_plan_search 的职责应该是“定位计划”
  • production_plan_detail 的职责应该是“提供计划内炉次安排、计划炉次数、配方摘要等详情证据”

一旦这层边界模糊，模型就会倾向于偷懒，直接在第一跳结果上作答。

Anthropic 的 tool use 文档里一直强调，工具描述不仅是“接口文档”，还是模型的行为边界。OpenAI 的 function calling 也是类似情况：它能强约束参数，但不会自动帮你定义“下一跳还该不该继续查”。

2. 把完成条件做成显式规则

这一步非常关键。

不要只告诉模型“如果证据够了就停”，而要告诉它：

  • 哪类问题需要哪类证据
  • 当前拿到的证据是否满足回答条件
  • 如果不满足，下一步应该去拿什么

比如对“数量/明细/安排/组成”类问题，可以定义一个通用原则：

  • 列表搜索工具只负责定位对象，不算最终回答证据
  • 只有详情工具、明细工具或汇总工具返回了目标字段，才允许结束回答

这样系统就不再依赖模型的主观判断，而是有了更稳定的完成条件。

3. 给 planner 一个结构化的中间状态

如果 planner 只有用户 query、历史消息和工具清单，它就只能靠自然语言去“猜”下一步。

更稳的做法是让 planner 内部至少形成这几个结构化状态：

  • intent
  • target_scope
  • required_evidence
  • missing_evidence

拿上面的例子来说，可以抽象成：

  • intent = plan_furnace_count
  • target_scope = today/current inherited plan scope
  • required_evidence = production_plan_detail.furnacePlanCount or furnacePlans
  • missing_evidence = detail_not_loaded

这样 planner 的行为就不再是“看起来差不多”，而是“缺哪份证据，就继续取哪份证据”。

4. 当某条链路已经稳定时，直接收敛成 facade tool

这一步不是所有系统一上来就要做，但在企业系统里通常是最终形态。

如果某类问题总是固定链路：

  • 先 search
  • 再 detail
  • 再 aggregate

那最稳的方式往往不是继续训练 planner，而是直接给它一个新的只读 facade tool。

比如：

  • production_plan_furnace_summary

内部自己完成：

  • 计划定位
  • 详情拉取
  • 炉次数统计
  • 结果摘要

这样对模型来说就只剩“选工具”和“补参数”，而不是“自己设计工作流”。

在工程上，应该怎么落到现有系统里

如果是已经有一套 Java MCP + Python Orchestrator 的系统，我更建议按下面这个顺序落。

第一步：给 tool 增加角色信息

不要只给工具名和 description，可以补一层更靠近 planner 的 metadata，例如：

  • role = locator
  • role = evidence
  • role = aggregate
  • satisfies = ["plan_detail", "furnace_count"]

有了这一层之后，planner 就能区分：

  • 哪些工具只是帮我找到对象
  • 哪些工具才是真正能支撑回答的证据源

第二步：在 planner 里引入“回答完成条件”

这是比补 few-shot 更关键的东西。

可以定义一组通用规则：

  • 用户问列表存在性，locator 结果可直接回答
  • 用户问对象内明细、数量、组成，locator 结果不能直接回答
  • 如果当前证据只到对象列表，必须继续调用 evidence 工具

这一步不需要一上来就上复杂 planning framework，但必须把“可停止”和“不可停止”从模型自由裁量中拿出来。

第三步：用例不要只测 final answer，要测事件链

很多多跳能力在联调里会被误判，是因为只看最终回答，不看中间事件。

更靠谱的验收应该是：

  • 有没有先调 locator
  • 有没有继续调 evidence
  • 最终回答是不是基于 evidence 收敛

也就是说，验收不只是看“答得像不像”，还要看“链路是不是对的”。

第四步：把高频稳定链路收敛成 facade

当某条链路已经很稳定，继续让 planner 每次都做多跳选择，其实是在浪费 token 和稳定性预算。

这时候就该把它沉淀成 facade：

  • 查询类问题优先做只读 facade
  • 写操作问题后面再考虑 preview/confirm/commit 三段式

这会比无休止地加 prompt 规则更稳。

一句更直接的总结

多跳规划能力增强的关键，不是让模型“更会猜下一步”，而是让系统把下面这几件事显式化：

  • tool 边界
  • 证据类型
  • 完成条件
  • 工作流归并

只要这四件事没立起来，模型就很容易在第一跳结果上过早停下。

反过来，只要这四件事被做成系统能力，即使用户换一种问法，planner 也更容易走出正确的 locator -> evidence -> summarize 链。
